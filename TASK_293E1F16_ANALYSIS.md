## ⏱️ 爬取速度变慢的原因分析

虽然任务最终失败了，但从日志可以分析出**如果成功**，速度也会很慢。以下是影响速度的关键因素：

### 1. **显式等待超时过长** ⏳ 最主要原因

**每次详情页抓取的时间分解**：

```
12:59:21.870 - 开始导航至详情页
12:59:22.402 - 开始等待关键元素（waitForSelector）
12:59:30.405 - ⚠️ 关键元素等待超时（8秒）
12:59:33.407 - 等待动态内容渲染（3秒固定延迟）
12:59:35.411 - 健康检查 + 提取数据
─────────────────────────────────────
总耗时: ~13.5秒/页
```

**时间分配**：
- **导航加载**: ~0.5秒
- **显式等待**: **8秒**（超时）← 最耗时
- **固定等待**: **3秒** ← 次耗时
- **数据提取**: ~0.5秒
- **总计**: **~13.5秒/页**

**问题**：
- 由于被反爬拦截，`waitForSelector` 等待8秒后超时
- 然后又固定等待3秒
- **每个详情页浪费11秒在无意义的等待上**

**如果未被拦截**（假设正常加载）：
- 显式等待: ~1-2秒（元素快速出现）
- 固定等待: 3秒（可优化）
- 总计: ~4-5秒/页

### 2. **并发度过高触发反爬** 🚀→🐌

**当前配置**: `concurrency: 5`

**实际表现**：

- 瞬时QPS = 5，超过智联招聘的阈值
- 触发反爬 → 所有请求返回错误页面 → 每个请求浪费13.5秒

**建议并发数**: **2-3**（降低60%）

### 3. **固定等待时间未动态调整** ⏲️

**当前代码**：

- 即使页面已完全加载，仍要等待3秒
- 无法根据实际加载情况动态缩短等待时间

**优化方案**：
- 如果 `waitForSelector` 成功，可减少固定等待至1秒
- 如果超时，才需要3秒补偿等待

### 4. **批次间延迟较长** 🔄

**当前配置**：

### 优先级2：优化等待策略 ⭐⭐

**当前代码问题**：
```typescript
// 1. 显式等待8秒（太长）
await page.waitForSelector('...', { timeout: 8000 });

// 2. 无论成功与否，都固定等待3秒

```

**优化方案**：

```typescript

  await page.waitForSelector('.summary-planes__title, .company-name', {
    timeout: 5000  // 从8秒降至5秒
  });

  // 如果快速加载成功，减少固定等待
```

**预期效果**：
- ✅ 正常页面：从13.5秒降至6秒（-55%）

### 优先级3：添加指数退避重试 ⭐

**当前行为**：遇到反爬立即失败，无重试

**优化方案**：

```typescript
let retryCount = 0;
const maxRetries = 2;

while (retryCount <= maxRetries) {
  try {
    await fetchJobDetailWithPage(page, jobUrl, basicInfo);
    break;  // 成功则退出
  } catch (error: any) {
    if (error.message.includes('PAGE_LOAD_FAILED') || 
        error.message.includes('ANTI_BOT')) {
      retryCount++;
      
      if (retryCount > maxRetries) {
        throw error;  // 达到最大重试次数，放弃
      }
      
      // 指数退避：2秒 -> 4秒
      const waitTime = Math.pow(2, retryCount) * 1000;
      this.log('warn', `[ZhilianCrawler] ⚠️ 检测到异常，${waitTime/1000}秒后重试 (${retryCount}/${maxRetries})...`);
      await new Promise(r => setTimeout(r, waitTime));
      
      // 刷新页面重试
      await page.reload({ waitUntil: 'domcontentloaded', timeout: 10000 });
    } else {
      throw error;  // 非反爬错误直接抛出
    }
  }
}
```

**预期效果**：
- ✅ 应对临时性反爬（如瞬时频率过高）
- ✅ 提高成功率约10-15%
- ⚠️ 会增加少量耗时（重试等待）

## 💡 根本原因总结

### 为什么会被反爬？

1. **并发度过高**：5个并发标签页同时发起请求
2. **请求频率过快**：批次间仅延迟3-4秒
3. **缺少随机性**：请求时间过于规律
4. **无会话维持**：未登录状态，被视为匿名爬虫

### 为什么速度慢？

1. **显式等待超时**：8秒超时（被拦截时完全浪费）
2. **固定等待过长**：无论加载情况都等待2-3秒
3. **无效重试**：被拦截后仍尝试提取，浪费时间
4. **并发失效**：虽然配置5并发，但全部失败，实际有效并发=0
