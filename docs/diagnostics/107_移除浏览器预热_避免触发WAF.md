## 问题现象

任务 `6cc19639-295f-46a2-97a6-e2f5ca910cac`（keyword=销售，city=北京，maxPages=1）运行 23 秒后结束，产出 0 条数据。

```
08:37:43 🔥 正在预热浏览器会话（第1次尝试）...
08:37:52 ⚠️ 预热页面过小(7884字节 < 50000)，疑似被WAF拦截，将重试...
08:37:52 ⏳ 等待5秒后重试预热...
08:38:00 🔥 正在预热浏览器会话（第2次尝试）...
08:38:00 ⚠️ 主页预热失败: Navigating frame was detached
08:38:03 ❌ 浏览器在预热后已断开连接，无法继续
08:38:04 任务结束，0条数据
```

## 根因分析

### 预热阶段的设计初衷

预热（warmup）在浏览器启动后、真实爬取前执行，目的是访问 51job 主页建立 cookie/session，让后续搜索请求看起来像是"已浏览过主页的正常用户"。

具体行为：
1. 创建临时页面
2. 访问 `https://we.51job.com/`（主页）
3. 等待 3 秒
4. 访问 `https://we.51job.com/pc/search?keyword=java&city=010000`（测试搜索）
5. 检查 HTML 大小是否 ≥ 50KB（正常页面阈值）
6. 失败则关闭页面，等 5-8 秒重试（最多 2 次）

### 预热为何适得其反

```
浏览器启动（全新 session、无 cookie）
  → 访问 51job 主页
  → 等待 3 秒
  → 访问 51job 搜索页
  → WAF 判定为自动化行为并返回拦截页面 (7884 字节)
```

**三个致命问题**：

**1. 行为模式暴露自动化特征**

正常用户不会在 3 秒内从主页跳到搜索页并只检查页面大小。这种模式（主页→搜索页快速跳转、无鼠标移动、无滚动行为）恰好是 WAF 检测自动化的典型信号。

**2. WAF 页面破坏浏览器状态**

被 WAF 拦截的页面（7884 字节）不是正常 HTML，关闭后浏览器 CDP 协议层处于不稳定状态：

```
预热页面 goto → 返回 WAF 拦截页
  → warmupPage.close()（静默关闭）
  → 浏览器残留未完成的 CDP 消息
  → 下一次 newPage() 或 goto()
  → "Navigating frame was detached"
  → browser.isConnected() → false
  → 任务终止
```

**3. 预热失败后无恢复路径**

之前的修复（`698daa6`）在预热失败后检查浏览器连通性，断开则退出。这个逻辑本身正确，但预热已经把浏览器拖入了无法恢复的状态 — 即使连通性检测通过，浏览器也背着 WAF 标记。

### 预热在最佳情况下的价值有限

即使预热成功（HTML ≥ 50KB），它建立的是从 51job 主页的 cookie。但实际爬取时，搜索页本身也会设置 cookie/session。预热提供的额外价值仅在于：

- 浏览器 cookie jar 中已有 51job 域名的 cookie（但搜索页也能设置）
- DNS 缓存已预热（但首次搜索页导航也会触发 DNS 解析）

这点价值远不足以抵消预热失败导致整个任务报废的风险。

## 修复方案

**直接移除预热阶段**，净删 57 行代码。

```typescript
// 修复前：60+ 行预热逻辑
const MIN_WARMUP_HTML = 50000;
let warmupSuccess = false;
for (let warmupAttempt = 0; warmupAttempt < 2 && !warmupSuccess; warmupAttempt++) {
  // ... 创建页面、导航、检测、关闭、重试 ...
}
if (!warmupSuccess) {
  // ... 浏览器健康检查、延迟恢复 ...
}

// 修复后：3 行初始化
await new Promise(r => setTimeout(r, 1000));
this.log('info', `🚀 浏览器就绪，直接开始爬取（跳过预热避免触发WAF）`);
```

### 搜索页自带的 WAF 防护（保留不变）

预热移除后，WAF 检测由搜索页加载流程中的 HTML 大小检查接管：

```
主搜索循环 → page.goto(搜索URL)
  → page.content() 获取 HTML
  → htmlLength < 50000 ?
    → 是：重载页面 → 仍小 → 尝试备用URL → 仍小 → 跳过AI判定为WAF
    → 否：正常提取数据
```

这个检测逻辑在每次搜索页加载时都会执行，比预热更精确（检测的是目标页面的实际响应，而非预热页面的状态）。

## 效果对比

| 指标 | 修复前（有预热） | 修复后（无预热） |
|------|----------------|-----------------|
| 启动到搜索的延迟 | 预热 15-30s + 恢复 5-10s | 1s 初始化延迟 |
| 预热触发 WAF 概率 | 高（主页+搜索页快速跳转） | 0（无预热） |
| 预热失败导致任务失败 | 是（浏览器损坏 → 退出） | 否 |
| 搜索页 WAF 检测 | 预热失败时不会到达 | 正常执行 |
| 浏览器稳定性 | 预热页面开闭引入 CDP 竞争 | 减少 2 次页面创建/销毁 |
