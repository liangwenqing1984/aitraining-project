## 问题现象

任务 `88e9875b-1c86-4115-b928-f5b35ef7e5a8`（keyword=销售，city=北京）。

```
Attempt 1:
  10:35:34 🏠 首页预访问 → 10:35:54 ✅ (20s)
  10:35:54 🔍 首页搜索: 找到 input[placeholder*="职位"] → 回车
  10:36:29 ⚠️ URL未变化！搜索失败（等待 35s 浪费）
  → 回退 page.goto(we.51job.com) → 7884 字节 WAF
  → 重载 WAF → 备用URL WAF → 回首页重试搜索
  10:37:17 ✅ 第二次搜索成功！跳转 we.51job.com/pc/search
  → HTML: 7884 字节 WAF（即使从首页自然跳转仍被封锁）
  → 0 条 → BROWSER_RESTART

Attempt 2:
  10:37:43 🏠 首页预访问 → [日志截断]
  
但 debug HTML 揭示真相: job51_page_1_recovered_1777545547992.html = 657KB 正常页面！
```

**矛盾**：第 3 次浏览器重启后的某次尝试成功获取了 657KB 正常搜索页，但仍产出 0 条数据。说明页面加载成功但数据提取失败。

## 根因分析

### 根因 1：SPA 选择器缺失 ← 最关键

debug HTML 分析揭示了 51job SPA 的真实 DOM 结构：

| 元素 | 实际类名 | 现有选择器 | 匹配？ |
|------|---------|-----------|--------|
| 职位卡片 | `.joblist-item` | `[class*="jobItem"]`, `[class*="job-item"]` | ❌ "joblist" ≠ "jobItem"/"job-item" |
| 职位名称 | `.jname`, `.joblist-item-jobname` | `[class*="title"]`, `[class*="name"]` | ❌ miss |
| 公司名称 | `.cname` | `[class*="company"]`, `[class*="cname"]` | ✅ `.cname` 匹配 |
| 薪资 | `.sal` | `[class*="salary"]`, `[class*="sal"]` | ✅ `.sal` 匹配 |
| 城市 | `.dc` | `[class*="city"]`, `.ltype` | ❌ miss |

`[class*="jobList"]` 中的驼峰 "jobList" 不匹配全小写 "joblist"（CSS 属性选择器区分大小写）。

**结果**：整个 `extractJobsFromDOM` 遍历全部 17 个选择器都无法匹配 `.joblist-item`，返回空数组 → 0 条数据。

### 根因 2：神策埋点数据未被利用

每个 `.joblist-item` 元素都有 `sensorsdata` 属性（神策数据埋点），内含**完整的 JSON 格式职位数据**：

```json
{
  "jobId": "171870110",
  "jobTitle": "CAD/CAM/PLM软件销售",
  "jobSalary": "1.5-3万",
  "jobArea": "北京·朝阳区",
  "jobYear": "1年及以上",
  "jobDegree": "大专",
  "companyId": "10335184",
  "jobType": "0",
  "funcType": "A0MP",
  "isPromote": "否"
}
```

这些数据覆盖了职位名称、薪资、城市区域、工作经验、学历、公司ID、jobId —— 比 DOM 解析更完整、更可靠。但代码从未尝试读取 `sensorsdata` 属性。

### 根因 3：首页搜索首次失败

首次调用 `searchViaHomepage` 时，找到输入框并按下回车，但 `waitForNavigation(30000)` 超时（35s 浪费），URL 未变化。而 1 分钟后第二次调用相同方法时却成功跳转。

原因推测：
- `waitForNavigation` 30s 超时过长（SPA 路由用 pushState，不会触发此事件）
- 51job 首页可能需要特定按钮点击（非回车键）
- 页面 JS 可能在首次搜索时尚未完全初始化
- 缺少 `form.submit()` 作为额外提交方式

### 根因 4：we.51job.com 仍被 WAF 封锁

即使首页搜索成功跳转（带完整 Referer、Cookie），we.51job.com 搜索结果页仍返回 7884 字节 WAF 页面。**IP 封锁已深入**。

但第 3 次浏览器重启后某次尝试获取到 657KB 正常页面，说明 **BROWSER_RESTART 机制最终有效**（新浏览器 + 新 userDataDir + 首页搜索链成功通过）。

## 修复方案

### 修复 1：sensorsdata 埋点数据提取（策略 S）← 最优先

在 `extractJobsFromDOM` 中新增最优先策略：

```typescript
// 策略 S：神策埋点 sensorsdata 属性
var sensorEls = document.querySelectorAll('[sensorsdata]');
for (var si = 0; si < sensorEls.length; si++) {
  var sd = JSON.parse(sensorEls[si].getAttribute('sensorsdata'));
  // 提取: jobId, jobTitle, jobSalary, jobArea, jobYear, jobDegree, companyId
  // 从同卡片 .cname 获取公司名称（sensorsdata 不含公司名）
  jobs.push({...});
}
if (jobs.length > 0) return jobs; // 成功后跳过所有 DOM 选择器遍历
```

**优势**：
- JSON 格式，100% 准确（无 DOM 解析误差）
- 包含 jobId → 可直接构造详情页链接 `jobs.51job.com/${city}/${jobId}.html`
- 包含 city、experience、education → 减少详情页抓取需求
- 包含 companyId → 可构造公司详情链接
- 不依赖 DOM 结构变化

**限制**：
- 不含公司名称（需从同卡片 `.cname` 补充）
- 依赖神策埋点不被移除

### 修复 2：补全 SPA 选择器

在策略 A（CSS 选择器）中新增：

```typescript
var selectors = [
  '.joblist-item', '[class*="joblist-item"]',  // 🆕 51job SPA
  'div.el', 'div.e', '.j_joblist > div',
  ...
];
```

选择器匹配更新：

```typescript
// 修复前
var titleEl = container.querySelector('[class*="title"], [class*="name"], ...');
var companyEl = container.querySelector('[class*="company"], [class*="cname"], ...');
var salaryEl = container.querySelector('[class*="salary"], [class*="pay"], ...');
var cityEl = container.querySelector('[class*="city"], [class*="area"], ...');

// 修复后（新增 SPA 类名优先）
var titleEl = container.querySelector('.jname, .joblist-item-jobname, [class*="jobname"], ...');
var companyEl = container.querySelector('.cname, [class*="cname"], ...');
var salaryEl = container.querySelector('.sal, [class*="salary"], ...');
var cityEl = container.querySelector('.dc, .joblist-item-jobinfo, .job-info, ...');
```

SPA waitForSelector 同步更新，新增 `.joblist-item`。

### 修复 3：优化首页搜索

**缩短超时**：`waitForNavigation` 从 30s → 15s（SPA 路由用 pushState，不会触发此事件）

**增加 form.submit() 提交**：回车键之外，额外触发表单的 submit 事件

**改进超时处理**：`waitForNavigation` 超时后先等 1.5s 检查 URL，再等 2s 二次检查（覆盖 SPA 路由延迟），而非直接判定失败
