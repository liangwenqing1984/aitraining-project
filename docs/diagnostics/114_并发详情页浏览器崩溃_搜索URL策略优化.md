# 并发详情页浏览器崩溃 + 搜索 URL 策略优化

## 涉及提交

| 提交 | 说明 |
|------|------|
| *待提交* | 主URL添加reportType=1；提前关闭搜索页；扩展互斥锁覆盖page.goto；首页搜索移出主路径 |

## 问题现象

任务 `a58ecee1-60c4-4d69-8c08-da2dabcef3a8`（keyword=销售，city=北京）产出了 20 条数据但质量严重下降：

```
10:51:57 🏠 首页预访问 → 10:52:10 ✅ (13s)
10:52:10 🔍 首页搜索 → 10:52:31 ⚠️ URL未变化（失败，21s浪费）
→ page.goto(url) → 7884 WAF
→ 重载 → 7884 WAF
10:53:13 备用URL (+reportType=1): 642186 字符 ✅ 正常！
10:53:23 XHR拦截: 20 条结构化数据 ✅
10:53:23 🚀 并发抓取详情页（3线程）
10:53:25 ❌ "Navigating frame was detached"
10:53:25 ❌ "Requesting main frame too early!"
10:53:29 ❌ "浏览器连接已断开"
→ 17/20 条降级为"使用列表数据"（无详情页字段）
10:54:13 任务完成，20条（但75%缺少详情信息）
```

## 根因分析

### 根因 1：首页搜索始终失败

51job 首页的搜索表单对自动化提交不响应。即使找到 `input[placeholder*="职位"]`、填入关键词、触发 `input`/`change`/`blur` 事件、按回车、调用 `form.submit()`——URL 始终停留在 `www.51job.com/`。

**结论**：首页搜索表单交互方案对 51job 不适用。51job 首页可能使用了更复杂的事件监听机制（如 React 合成事件、或需要特定鼠标事件序列）。

### 根因 2：reportType=1 是真正的 WAF 绕过参数

经过多次任务对比：

| URL 参数 | HTML 大小 | 结果 |
|----------|----------|------|
| `keyword=...&city=...&page=1` | 7884 字节 | WAF |
| `keyword=...&city=...&page=1&pageSize=20` | 7884 字节 | WAF（旧版触发） |
| `keyword=...&city=...&page=1&reportType=1` | 642,186 字节 | **✅ 正常** |
| `keyword=...&jobArea=...&page=1&searchType=2` | 7884 字节 | WAF（首页搜索跳转） |

`reportType=1` 是绕过 WAF 的关键参数。它是 51job 的合法搜索参数（report 类型），WAF 将其视为"报表/导出类请求"从而放行。

### 根因 3：并发详情页抓取导致浏览器崩溃

详情页抓取使用并发模式（config.concurrency=3），同时打开 3 个新页面。问题链条：

1. **搜索页未关闭**（open + request interception 活跃）
2. **3 个新页面并发创建**（即使有 mutex 保护 newPage，goto 仍并发执行）
3. **CDP 协议竞争**："Navigating frame was detached" + "Requesting main frame too early!"
4. **浏览器进程崩溃**："浏览器连接已断开"
5. **后续 17/20 条全部降级**为列表数据

根本问题：
- 搜索页占用 1 个 tab + request interception
- 3 个详情页同时 newPage + goto → 浏览器负荷过重
- 互斥锁仅保护 `newPage()`，`page.goto()` 仍然并发执行
- Chrome 在 4 个 tab 同时活跃时 CDP 协议不稳定

## 修复方案

### 修复 1：主 URL 直接添加 reportType=1

```typescript
// 修复前
const url = `https://we.51job.com/pc/search?keyword=...&city=...&page=${currentPage}`;

// 修复后
const url = `https://we.51job.com/pc/search?keyword=...&city=...&page=${currentPage}&reportType=1`;
```

跳过首页搜索尝试，直接使用已被证明有效的参数组合。节省了约 20s 首页搜索失败等待时间。

### 修复 2：提前关闭搜索页

在开始详情页抓取前关闭搜索页：

```typescript
// === 关闭搜索页释放浏览器资源 ===
try { await page.close(); } catch { /* ignore */ }

// === 处理职位：列表数据 + 详情页抓取 ===
const concurrency = config.concurrency || 1;
```

确保详情页抓取时浏览器只有正在使用的详情页 tabs，不会有搜索页的 request interception 干扰。

### 修复 3：互斥锁扩展到 page.goto()

```typescript
// 修复前：仅保护 newPage()
const currentMutex = this.pageCreateMutex;
await currentMutex;
try {
  page = await browser.newPage();  // 序列化
} finally {
  releaseMutex!();                 // 过早释放
}
await page.goto(jobUrl, ...);      // 仍然并发！

// 修复后：保护 newPage + fingerprint + goto
const currentMutex = this.pageCreateMutex;
await currentMutex;
try {
  page = await browser.newPage();       // 序列化
  await this.setupPageFingerprint(page); // 序列化
  await page.setRequestInterception(true);
  page.on('request', ...);              // 序列化
  await page.goto(jobUrl, ...);         // 序列化 ← 关键！
} finally {
  releaseMutex!();                       // goto 完成后释放
}
```

互斥锁现在覆盖 `newPage()` + `page.goto()` 两个浏览器密集操作。`page.evaluate()`（CPU 提取）在释放锁后并发执行。这样：
- 浏览器操作被序列化，避免 CDP 竞争
- 数据提取仍可并发，保持吞吐量
- 每个时刻浏览器只有 1 个 page 在执行网络请求

### 修复 4：WAF 恢复 URL 反向策略

由于主 URL 现在包含 `reportType=1`，备用 URL 改为**去掉**该参数：

```typescript
// 修复前：添加 reportType=1（但主 URL 已有，会重复）
const altUrl = url + '&reportType=1';

// 修复后：去掉 reportType=1，恢复到原始 URL
const altUrl = url.replace('&reportType=1', '').replace('?reportType=1&', '?').replace('?reportType=1', '');
```

同样的策略应用于 AI 恢复路径（原 `&pageSize=10` → 改为去掉 `reportType=1`）。
