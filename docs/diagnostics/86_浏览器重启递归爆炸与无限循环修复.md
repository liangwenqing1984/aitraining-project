## 一、问题现象

任务 `5145e61d`（25 关键词 × 1 城市）在第 5 个组合后触发主动浏览器重启，随即进入死循环，连续重启 8 次仍无法继续：

...（几秒后）
12:48:12  🔄 已处理 5 个组合，主动重启浏览器... ← 再次触发！
```

堆栈跟踪显示每次重启增加一层递归，8 轮后达到 10 层：

```

## 二、根因分析

### Bug 1 — 无限重启循环（zhilian.ts）

主动重启使用绝对组合编号判断：

```typescript
// 修复前
const COMBINATIONS_PER_BROWSER = 5;
if (currentCombination % COMBINATIONS_PER_BROWSER === 0 && ...) {
  throw restartError;  // combinationIndex = 5
}
```

断点续传从组合 5 恢复后，`currentCombination = 5`，`5 % 5 === 0` 条件立即成立，又触发重启 → 回到组合 5 → 无限循环。

### Bug 2 — 递归爆炸（taskService.ts）

重启逻辑用递归自调用实现：

```typescript
// 修复前 — taskService.ts:555
await this.executeCrawling(taskId, configWithResume, controller, resumeState, logger);
return;
```

每次重启都是对自身的递归调用，JS 不保证尾调用优化，导致调用栈持续增长。与 Bug1 循环结合后，短期内即达 10 层以上。

---

## 三、修复方案

### 修复 1 — 基于重启后增量判断（zhilian.ts:1662-1665）

```typescript
const COMBINATIONS_PER_BROWSER = 5;
const combosSinceRestart = currentCombination - Math.max(0, startCombinationIndex);
if (combosSinceRestart > 0 && combosSinceRestart % COMBINATIONS_PER_BROWSER === 0 && ...) {
```

`startCombinationIndex` 是断点续传起点。断点续传从组合 5 开始：
- 组合 5：`combosSinceRestart = 5 - 5 = 0` → 不触发 ✓
- 组合 10：`combosSinceRestart = 10 - 5 = 5` → 触发重启 ✓

### 修复 2 — 重启次数上限（taskService.ts:483-490）

```typescript
progressInfo.restartCount = (progressInfo.restartCount || 0) + 1;
const MAX_RESTARTS = 10;
if (progressInfo.restartCount > MAX_RESTARTS) {
  throw new Error(`浏览器重启次数超过上限(${MAX_RESTARTS})，任务终止`);
}
```

## 四、效果对比

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 从组合 5 断点续传 | 立即触发重启 → 死循环 | combosSinceRestart=0，不触发，继续爬取 |
| 连续爬完 10 个新组合 | 正常工作 | 第 5、10 个各触发一次适度重启 |
| 异常连续重启 | 无限递归直至 OOM | 第 10 次后终止，标记任务失败 |
