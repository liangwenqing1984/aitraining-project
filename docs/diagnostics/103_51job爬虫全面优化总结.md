## 一、SPA 感知加载（核心架构重写）

### 问题
51job 搜索页是 Vue.js SPA，初始 HTML 仅几十字节的 JS 加载壳。旧代码用 `waitUntil: 'load'` 永远等不到内容。

### 方案
- `page.goto` 改用 `waitUntil: 'networkidle2'`（等 SPA JS 执行完毕 + 无网络活动 500ms）
- `page.waitForSelector` 等待职位列表 DOM 出现（15s 超时）
- XHR 响应拦截：捕获 51job 内部 API (`/api/job/search-pc`) 返回的 JSON 结构化数据
- 滚动触发懒加载 + 保存调试 HTML 快照

### 数据提取三层策略

| 优先级 | 策略 | 数据来源 |
|--------|------|---------|
| 1 | XHR 拦截 | API JSON 响应 → 结构化映射 |
| 2 | DOM 选择器 | 已知 CSS 选择器遍历职位卡片 |
| 3 | AI 选择器 | LLM 推荐新选择器（适应站点改版） |

## 二、AI 反爬集成

### 三个接入点

| 接入点 | 触发时机 | 调用的 LLM 方法 |
|--------|---------|----------------|
| 页面加载后 | 每次搜索页加载完 | `classifyPage(html, url)` 检测反爬类型 |
| 选择器失效 | DOM 提取返回 0 结果 | `suggestSelectors(html, '职位列表')` 推荐新选择器 |
| 反爬检测 | 页面非 normal 类型 | `recommendAction(classification)` 获取应对策略 |

### 页面分类类型

```
normal → 正常采集
captcha → 等 AI 推荐时长后重试
waf → 页面重载
empty → 页面重载
error → 页面重载
login → 等 AI 推荐时长后重试
```

### 连续反爬退避
连续 3 页触发反爬 → 退避时间按 `30s × 2ⁿ` 递增（上限 5 分钟）

## 四、page.evaluate 的 `__name` 错误修复

### 问题演化

```
阶段1（原始代码）
  const get = (sels: string) => { ... }
  → esbuild 注入 __name(get, "get")
  → 浏览器: ReferenceError: __name is not defined

阶段2（第一轮修复 ff1c1ef — 无效）
  var get = function(sels) { ... }
  → esbuild 仍然注入 __name(get, "get")
  → 错误依旧

阶段3（第二轮修复 92663ae — 彻底解决）
  function get(sels) { ... }
  → 函数声明 — esbuild 唯一不注入 __name 的形式
  → 错误消失
```

### 根因
`tsx` 底层 esbuild 默认 `keepNames: true`，对任何变量赋值的函数表达式都注入 `__name` 辅助函数。只有函数声明（Function Declaration）可以免疫。

### 影响范围：4 个 evaluate 回调全部去 TS 化

| 方法 | 回调类型 | 关键修复 |
|------|---------|---------|
| `fetchJobDetail` | `page.evaluate` | `function get(sels) {}` 声明 + 匿名回调 |
| `setupPageFingerprint` | `evaluateOnNewDocument` | `(window as any)` → `window` |
| AI 选择器 | `page.evaluate` | `(selector: string)` → `(selector)` |
| `extractJobsFromDOM` | `page.evaluate` | `as HTMLAnchorElement` → `&&` guard |

## 五、并发详情页抓取

### 修复前
```typescript
// 硬编码串行 for 循环，忽略 config.concurrency
for (let i = 0; i < pageJobs.length; i++) {
  jobData = await this.fetchJobDetail(browser, url, job, config);
  yield jobData;
  await this.randomDelay(1500, 3500);  // 每条间隔
}
```

### 修复后
```typescript
const concurrency = config.concurrency || 1;
const maxConcurrency = Math.min(concurrency, 3);  // 上限3

if (maxConcurrency <= 1) {
  // 串行模式（原有逻辑）
} else {
  // 并发模式：Promise.allSettled 批次并行
  for (let batchStart = 0; batchStart < pageJobs.length; batchStart += maxConcurrency) {
    const batchPromises = batch.map(async (job, idx) => { ... });
    const results = await Promise.allSettled(batchPromises);
    // 按序 yield 结果
    await this.randomDelay(2000, 4000);  // 批次间延迟
  }
}
```

| 特性 | 修复前 | 修复后 |
|------|--------|--------|
| 处理模式 | 硬编码串行 | `concurrency≤1` 串行 / `>1` 并发 |
| config.concurrency | 被忽略 | 生效（上限 3） |
| 错误隔离 | 无 | `Promise.allSettled` |
| 预期速度 | ~45s/条 | ~7-15s/条（3并发） |

## 六、WAF 拦截防御

### 问题
51job WAF 拦截返回 ~7,884 字节小页面，AI 容易假阳性（confidence=0.95 判为 normal）。

### 根因链
```
预热拿到 WAF 小页面 → 未验证 → 搜索时坏 session
→ 搜索页也是小页面 → AI 假阳性 normal → 选择器失效
→ AI 返回空选择器 "" → querySelectorAll('') 抛异常 → 0 jobs
```

### 三层防御

| 防御 | 阈值 | 行为 |
|------|------|------|
| 预热验证 | HTML < 50KB | 等 5-8s 后重试 1 次，两次均失败报警 |
| 搜索页 WAF 检测 | HTML < 50KB | 跳过 AI 直接判 WAF → reload → 备用 URL |
| AI 空选择器保护 | `selector.trim() === ''` | 跳过而非抛 `SyntaxError` |

正常 51job 搜索页 HTML 约 500KB-650KB，50KB 阈值提供 10 倍安全边际。

## 七、浏览器稳定性

### 问题
3 个并发 `browser.newPage()` 无互斥锁 → Chrome CDP 协议 `Target.createTarget` 竞争 → 浏览器崩溃。

### 修复：页面创建互斥锁

```typescript
// 类属性
private pageCreateMutex: Promise<void> = Promise.resolve();

// fetchJobDetail 中
const currentMutex = this.pageCreateMutex;
let releaseMutex: () => void;
this.pageCreateMutex = new Promise<void>(resolve => { releaseMutex = resolve; });
await currentMutex;
try {
  page = await browser.newPage();
} finally {
  releaseMutex!();
}
```

同一时刻仅一个 `browser.newPage()` 调用在执行。

### 辅助措施

| 措施 | 位置 | 机制 |
|------|------|------|
| 批次内错峰延迟 | 并发 batch.map 内 | 第 2/3 个任务 delay 800-2000ms |
| 浏览器前置检查 | batch 任务 try 前 | `browser.isConnected()` 为 false → 直接降级 |
| 首次失败日志 | fetchJobDetail catch | 非最终重试也记录 warn |
