# 浏览器崩溃恢复 Excel 二进制误读导致 record_count 虚高

**日期**: 2026-04-28  
**影响范围**: 任务进度计数、数据一致性、断点续传  
**关联文件**: `taskService.ts`  
**受影响任务**: `938a0a93-0d70-4fd8-b524-cc2db9daf1b9`（多职位-单城市）

## 现象

任务 `938a0a93` 完成后界面显示 82 条记录，但 Excel 文件实际只有 31 行数据。差值 51 条。

### 3. 关键发现

日志中显示 `从CSV文件读取 56 条`，但此时实际上 UI 关键词只成功写入了 5 条。**56 远大于 5**。

## 根因

崩溃恢复代码 [taskService.ts:448-453](code/backend/src/services/taskService.ts#L448-L453)：

```typescript
const fileContent = fs.readFileSync(filepath, 'utf-8');  // ← BUG: .xlsx 二进制被当文本读！
const lines = fileContent.split('\n').filter(line => line.trim().length > 0);
initialRecordCount = Math.max(0, lines.length - 1);
```

`.xlsx` 是 ZIP 压缩的二进制格式，用 UTF-8 文本方式读取会产生随机的换行符分布。13KB 的 Excel 文件被误读为 57 个"文本行"（56 条数据 + 1 行表头），这个数字完全是二进制垃圾的随机结果。

## 时序复现

| 步骤 | 事件 | Excel实际行数 | totalRecords |
|------|------|:----------:|:----------:|
| 1 | 启动，爬 "UI" 关键词 → 写入 5 条 | 5 | 5 |
| 2 | 浏览器崩溃 | 5 | — |
| 3 | 🔴 崩溃恢复：读 .xlsx 为文本 → 得 **56 行垃圾** | 5 | **56** |
| 4 | 续传 "Java" → 写入 17 条 | 22 | 56+17=73 |
| 5 | 续传 "Python" → 写入 9 条 | 31 | 73+9=82 |
| 6 | 任务完成，`record_count = 82` | **31** | **82** |

**虚高值 = 56（二进制垃圾） - 5（实际） = 51 条**

## 修复

根据文件扩展名选择读取方式：

```typescript
const ext = path.extname(filepath).toLowerCase();
if (ext === '.xlsx') {
  // 用 ExcelJS 读取实际行数
  const workbook = new ExcelJS.Workbook();
  await workbook.xlsx.readFile(filepath);
  const worksheet = workbook.worksheets[0];
  initialRecordCount = Math.max(0, worksheet.rowCount - 1);
} else {
  // .csv 文件用文本方式读取
  const fileContent = fs.readFileSync(filepath, 'utf-8');
  const lines = fileContent.split('\n').filter(line => line.trim().length > 0);
  initialRecordCount = Math.max(0, lines.length - 1);
}
```
