## 一、停止→继续生成重复 Excel（四层根因）

### 第一层：catch 块 throw/break 跳过断点保存

**问题**：while 循环内部的 try-catch 中，`throw error` 和 `break` 都跳过了 line 1711 的唯一 `_resumeState` 保存点。

**修复** (`e39d265`)：catch 块简化为两个分支，在 throw/break 之前先保存 `_resumeState`。

### 第二层：throw 导致 generator 异常退出

**问题**：虽然保存了 `_resumeState`，但 `throw error` 仍然导致 generator 异常退出 → taskService 的 for-await 收到异常 → `isBrowserCrash=false` → 任务标为 **failed**（而非 stopped）→ stop handler 不执行 → `record_count` 不更新。

**修复** (`711e723`)：catch 块重构为**三路分支**：

```typescript
if (this.checkAborted()) {
    保存 _resumeState → hasNextPage = false;  // 不抛错，自然退出
} else if (isBrowserCrash || isDetachedFrame) {
    throw crashError;  // 真正的崩溃→重试
} else {
    break;  // 非崩溃错误→跳过当前页
}
```

中止路径通过 `hasNextPage=false` 让 while 循环自然退出，generator 正常结束，stop handler 正确执行。

### 第三层：randomDelay 期间的断点保存缺口

**问题**：代码结构如下：

```
while (hasNextPage && !checkAborted()) {
    try { ... } catch { ... }
    if (checkAborted() || !hasNextPage) { 保存→break; }  // line 1711
    currentPage++;                          // line 1729
    await this.randomDelay(2000, 4000);     // line 1736 ← 在此中止...
}
// line 1741 ← ...断点未保存！
```

如果用户在 page 间延迟期间点击停止，`abortableSleep` 中断，while 条件失效退出，但 `_resumeState` 未被保存。

**修复** (`461fea4`)：在 line 1741 的 `checkAborted()` 分支增加防御性保存（仅当 `_resumeState` 不存在时写入）。

### 第四层：`record_count` camelCase 导致计数归零

**问题**（最关键）：DB 封装层 `database.ts` 的 `convertToCamelCase` 将 `record_count` → `recordCount`。但 `taskService.ts` line 154 使用 snake_case：

```typescript
initialRecordCount = existingTask.record_count || 0;  // ❌ 永远是 undefined → 0
```

导致：
- Excel 正确复用 ✅（`csvPath` 无下划线不受影响）
- 但 `起始记录数` 始终显示为 0 ❌
- `totalRecords` 从 0 开始计数 ❌
- 用户看到计数从 1 开始 = 等同于"生成了新 Excel"

**修复** (`5d5e236`)：`existingTask.record_count` → `existingTask.recordCount`。

## 二、反爬拦截检测与重试

### 问题

智联招聘使用 **GeeTest 反爬系统**。检测到异常访问后返回空 body 的 SPA 壳：

```html
<title>「Java招聘 2026年哈尔滨Java招聘信息」-智联招聘</title>  ← 标题正常
<link rel="dns-prefetch" href="//dkapi.geetest.com">           ← GeeTest
<meta content="{加密token}">                                   ← 反爬challenge
<!-- body 完全为空，bodyLength=0 -->
```

此前 bodyLength=0 的组合被标记为"完成"（0 条数据），**数据永久丢失**。

任务 `31c64fd2` 中 62 个关键词有 6 个因此丢失：Java、人工智能、数据开发、测试、电气工程师、电子工程师。

关键证据：电子工程师首次 bodyLength=0，浏览器重启重试后 bodyLength=193535，成功提取 14 条。

### 修复 (`bd82754`)

`waitForSelector` 超时 + `bodyLength === 0` → 判定为反爬拦截：

1. 保存 `_resumeState` 断点
2. 抛出 `BROWSER_CRASH_RECOVERABLE` 错误
3. 触发 taskService 的浏览器重启+重试机制
4. `isBrowserCrash` 识别列表加入 `BROWSER_CRASH_RECOVERABLE`

| 修改 | 文件 | 行 |
|------|------|-----|
| 反爬检测+重试 | `zhilian.ts` | ~439-471 |
| `isBrowserCrash` 扩展 | `zhilian.ts` | ~1602 |

## 三、csv_files 表重复记录

### 问题

停止/完成/失败三个路径都直接 `INSERT INTO csv_files`，每次停止→继续→停止都产生新记录。任务 `31c64fd2` 经过 3 次停止，csv_files 表有 3 条记录指向同一个物理文件。

### 修复 (`a3f6258`)

三个路径改为 upsert：先 `SELECT` 检查 `task_id` 是否存在，存在则 `UPDATE`，不存在才 `INSERT`。
