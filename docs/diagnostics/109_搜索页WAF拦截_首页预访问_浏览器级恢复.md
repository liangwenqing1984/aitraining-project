## 根因分析

### WAF 如何检测自动化

即使已启用 puppeteer-extra + stealth 插件（隐藏 `navigator.webdriver`、伪造 plugins/chrome 对象），Aliyun WAF 仍能通过**行为模式**检测自动化：

```
浏览器启动（全新 session、无 cookie、无历史）
  → 直接导航到搜索 API 页（带复杂查询参数）
  → WAF 判定：无首页浏览 → 无 cookie 积累 → 非人类行为
  → 返回滑动验证码页面（7884 字节）
```

**关键信号**：

| 信号 | 表现 | WAF判定 |
|------|------|---------|
| 无浏览历史 | 首次请求就是搜索 API | 高概率自动化 |
| 无 cookie | 空 cookie jar | 新访客 + 搜索 = 爬虫 |
| 直接深度链接 | 跳过了首页→搜索的自然路径 | 人类不会这样做 |
| 请求时序 | 启动后 ~3s 即发搜索请求 | 太快，不自然 |

### 之前已移除的预热为何不足以解决此问题

之前的 commit `7a0d3a9` 移除的预热方案有以下致命缺陷：

1. **创建临时页面 → 关闭**：首页和搜索页在不同页面中，关闭预热页面后 cookie 丢失
2. **快速跳转**：3 秒内从首页跳搜索页，反而强化了自动化信号
3. **页面开闭竞争**：`warmupPage.close()` 后立即 `browser.newPage()` 引发 CDP 竞争崩溃

### 当前为何没有恢复机制

检测到 WAF 后的处理流程：
```
HTML 过小 → 重载 → 仍小 → 备用URL → 仍小
  → AI 确认为 WAF → "已在HTML大小检测中处理过WAF，跳过AI重载流程"
  → 0 条职位 → break → 组合结束 → 任务结束
```

没有将 WAF 检测传递到 taskService 层的机制。浏览器没有被重启，只是放弃了当前页。

## 修复方案：双管齐下

### 方案 1：首页预访问建立会话（预防）

在首次搜索前，使用**同一页面**先访问 51job 首页，模拟人类自然浏览路径：

```
浏览器启动
  → 创建页面
  → 设置指纹
  → 🏠 访问 51job 首页（www.51job.com）
  → 等待 3-7 秒（随机）
  → 缓慢滚动 200-700px
  → 等待 1.5-3.5 秒（随机）
  → ✅ 会话建立，cookie 已写入浏览器 jar
  → 设置资源拦截（阻止图片/字体加速后续请求）
  → 导航到搜索 URL
  → 正常提取数据
```

**与已移除的预热的区别**：

| 维度 | 旧预热（已移除） | 新首页预访问 |
|------|-----------------|-------------|
| 页面 | 创建独立临时页面，用完关闭 | 复用搜索页面，不创建/关闭额外页面 |
| cookie 传递 | 随页面关闭丢失 | 保留在同一 browser context 中 |
| 访问内容 | 首页 + 搜索页（2个URL） | 仅首页（1个URL） |
| 延迟节奏 | 固定 3 秒 | 随机 3-7 秒 + 滚动 + 1.5-3.5 秒 |
| 人类模拟 | 无交互 | 有滚动行为 |
| CDP 竞争 | 有（页面创建/关闭） | 无（单页面连续操作） |

```typescript
if (!sessionInitialized) {
  this.log('info', `🏠 首次访问，先浏览51job首页建立会话...`);
  try {
    await page.goto('https://www.51job.com/', { waitUntil: 'networkidle2', timeout: 30000 });
    await new Promise(r => setTimeout(r, 3000 + Math.random() * 4000));
    await page.evaluate(function() {
      window.scrollTo(0, 200 + Math.floor(Math.random() * 500));
    });
    await new Promise(r => setTimeout(r, 1500 + Math.random() * 2000));
    this.log('info', `✅ 会话初始化完成，cookie已建立`);
  } catch (homeErr: any) {
    this.log('warn', `⚠️ 首页访问失败(${homeErr.message})，继续尝试搜索...`);
  }
  sessionInitialized = true;
}
```

失败不阻塞：首页访问失败时仍继续尝试搜索（可能直接搜索也能通过）。

### 方案 2：浏览器级 WAF 恢复（治疗）

如果首页预访问后搜索页仍被 WAF 拦截且整个会话产出 0 条数据，触发 taskService 的 `BROWSER_RESTART_SCHEDULED` 机制重启浏览器：

```
会话结束
  → totalYielded === 0 && wafDetected === true ?
    → 是：throw BROWSER_RESTART_SCHEDULED
      → taskService 捕获
      → 关闭旧浏览器
      → 创建新浏览器（新 userDataDir）
      → 重新调用 crawl()
    → 否：正常结束
```

```typescript
// 在 crawl() 方法末尾
if (totalYielded === 0 && wafDetected) {
  const restartErr = new Error('BROWSER_RESTART_SCHEDULED: 检测到WAF拦截，零数据产出');
  (restartErr as any).shouldRestart = true;
  throw restartErr;
}
```

taskService 中已有对应的处理逻辑（`isPlannedRestart` 检测 + 重启计数限制 + 退避延迟），无需修改。
