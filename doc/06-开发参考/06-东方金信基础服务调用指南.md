# 基础服务调用指南

接入基础服务平台的应用需要有自己的 `appCode`、`moduleCode`，在调用相关能力时会用到。

其中：

- `appCode` 为应用编号，每个应用只会有一个 `appCode`
- `moduleCode` 为应用中每个模块的编号，一个应用可能会有多个 `moduleCode`

这两个值都需要开发人员在创建项目时事先指定好，并保存到项目中。

本文档整理当前示例项目已接入的基础服务能力，重点覆盖两类场景：

- BASE 定时任务组分页与操作
- BASE 工作流提交、待办与流程查询

文档优先记录当前示例项目中已经落地的调用方式，便于后续继续扩展其他基础服务模块。

相关接口说明文档：

- Auth 服务接口说明：[02-Auth服务接口说明.md](./02-Auth服务接口说明.md)
- Base 服务接口说明：[03-Base服务接口说明.md](./03-Base服务接口说明.md)

## BASE 定时任务组调用整理

当前项目已新增“BASE基础服务定时任务”页面，入口位于首页功能区。

### 1. 任务组分页列表

分页查询调度任务组使用以下接口：

```http
GET /job/group/pageJobGroup
```

当前前端调用方式：

- 请求方法：`GET`
- 参数位置：query string
- 登录态：浏览器自动透传 cookie，`fetch` 需带 `credentials: "include"`
- 固定筛选条件：`appCode = {APP_CODE}`
- 当前分页参数：
  - `pageNum = 当前页 + 1`
  - `pageSize = 10`

示例中的 `APP_CODE` 表示从应用配置中读取的实际 `appCode`。

当前项目中的请求示例：

```ts
const query = new URLSearchParams({
  appCode: APP_CODE,
  pageNum: String(pageNumber + 1),
  pageSize: String(pageSize)
});

const response = await fetch(`${baseHost}/job/group/pageJobGroup?${query.toString()}`, {
  credentials: "include",
  headers: {
    Accept: "application/json"
  }
});
```

根据 [03-Base服务接口说明.md](./03-Base服务接口说明.md)，该接口返回类型为 `PageResult<JobGroupApiVO>`，当前项目实际按以下分页字段读取：

- `pageNum`
- `pageSize`
- `total`
- `data`

前端当前展示字段包括：

- `jobGrpId`
- `jobGrpName`
- `appCode`
- `moduleCode`
- `creatorName`
- `createTm`
- `modifyTm`

当前前端字段映射约定：

1. 行主键优先使用 `jobGrpId`，在前端统一转成 `string`
2. “任务组名称”优先读取 `jobGrpName`
3. `creatorName` 若接口未直接返回，页面允许回退显示为 `creator` 或其他兼容字段
4. 页码在前端内部按 `0` 基处理，请求 BASE 时再转回 `1` 基
5. 当前列表示例只查询指定 `appCode` 下的任务组数据

### 6. 创建调度任务

当前项目中的“创建调度任务”入口位于定时任务页面右上角，点击后会打开悬浮创建表单。

当前创建能力使用以下接口：

```http
POST /job/group/addSimpleJobGroup
```

根据 [03-Base服务接口说明.md](./03-Base服务接口说明.md)，该接口请求体类型为 `JobSimpleGroupApiInput`，包含：

- `jobGrpName`
- `appCode`
- `moduleCode`
- `isPreset`
- `isGroupStart`
- `groupBusinessType`
- `jobInfoApiInputs`

当前项目中的调用方式：

```ts
await fetch(`${baseHost}/job/group/addSimpleJobGroup`, {
  method: "POST",
  credentials: "include",
  headers: {
    Accept: "application/json",
    "Content-Type": "application/json"
  },
  body: JSON.stringify({
    jobGrpName,
    appCode: APP_CODE,
    moduleCode,
    isPreset: "0",
    isGroupStart: true,
    groupBusinessType,
    jobInfoApiInputs: [
      {
        jobName,
        jobCode,
        jobCron,
        jobDescribe,
        jobParam,
        alarmEmail,
        requestUrl,
        requestType,
        jobType,
        requestCorrectMark,
        jobTag,
        appCode: APP_CODE,
        moduleCode,
        jobBusinessType
      }
    ]
  })
});
```

当前前端表单约定如下：

1. 创建入口按钮文案为“创建调度任务”
2. `appCode` 由应用配置提供，不要在页面中写死
3. `moduleCode` 由用户填写
4. 当前默认 `isPreset = "0"`
5. 当前默认 `isGroupStart = true`
6. `Cron` 默认值为 `0 */5 * * * ?`
7. 当前表单中带 `*` 的任务字段为必填：
   - `任务名称`
   - `任务编码`
   - `Cron 表达式`
   - `请求地址`

当前页面不再采用旧的“两步创建”方式：

- 旧方案：先 `/job/group/add`，再 `/job/add`
- 当前方案：直接调用 `/job/group/addSimpleJobGroup`

### 7. 当前页面交互

当前项目中的定时任务组页面交互如下：

1. 首页点击 `BASE基础服务定时任务`
2. 跳转到 `#/base-scheduler-groups`
3. 页面进入后调用 `GET /job/group/pageJobGroup` 获取任务组分页列表，并带上当前应用配置的 `appCode`
4. 页面右上角支持“创建调度任务”入口，提交时调用 `POST /job/group/addSimpleJobGroup`
5. 列表中支持以下操作：
   - 启动：`POST /job/group/start?grpId=...`
   - 停止：`POST /job/group/stop?grpId=...`
   - 测试执行：`POST /job/group/startSchJobGroup?jobGroupId=...`
   - 删除：`POST /job/group/delete?grpId=...`
6. 创建成功后刷新任务组第一页
7. 启动、停止、测试执行、删除成功后刷新当前分页列表
8. 成功或失败结果统一通过页面右上角悬浮框提示

### 8. 接入建议

后续如果继续扩展 BASE 定时任务相关功能，建议遵循以下约定：

1. 任务组列表、任务组操作和创建能力都直接调用 BASE，不额外在本项目后端包一层
2. 列表中的所有任务组 ID 在前端统一按 `string` 处理
3. 对于参数名不一致的接口要单独记录：
   - `start / stop / delete` 使用 `grpId`
   - `startSchJobGroup` 使用 `jobGroupId`
4. 查询任务组列表时建议统一追加当前应用配置的 `appCode`，避免在页面逻辑中写死
5. 操作结果优先使用悬浮提示，不要把错误信息直接插入表格上方打断阅读
6. 若继续扩展创建表单，可优先补充：
   - `isPreset`
   - `isGroupStart`
   - `groupBusinessType`
   - `jobBusinessType`
7. 如果后续增加搜索条件，可优先接入 `keyword`、`moduleCode`

## 1. 接口概览

发起工作流实例使用以下接口：

```http
POST /workflow/submit
```

请求返回的 `data` 为工作流实例 ID，后续可基于该 ID 查询工作流状态、审批节点和审批结果。

## 2. 调用前提

调用 BASE 工作流接口前，建议先确认以下条件：

- 已拿到当前登录用户的 `access_token`
- 请求中可透传登录态：
  - `Authorization: Bearer {access_token}`
  - `Cookie: seabox_acc_token={access_token}; access_token={access_token}`
- 已明确当前业务对应的：
  - 工作流定义 ID `wfBaseDefFileId`
  - 审批对象 ID `approvalObjectId`
  - 审批对象标题 `approvalObjectTitle`
  - 审批对象类型 `approvalObjectType`
  - 工作流表单参数 `approvalFormData`

## 3. 请求入参

请求体类型：`SubmitWfInstanceInput`

字段说明：

- `wfBaseDefFileId` (Long)
  - 工作流定义标识
- `approvalObjectId` (String)
  - 审批对象标识
  - 一般存业务主键，建议统一按字符串传递
- `approvalObjectTitle` (String)
  - 审批对象标题
  - 建议传用户可读标题，便于审批列表识别
- `approvalObjectType` (String)
  - 审批对象类型
  - 建议按业务域约定固定编码，便于后续扩展
- `approvalFormData` (List<SubmitWfInstanceFormDataInput>)
  - 工作流表单参数列表

其中 `SubmitWfInstanceFormDataInput` 结构如下：

```json
{
  "wfParamName": "参数英文名",
  "wfParamCnName": "参数中文名",
  "wfParamValue": "参数值"
}
```

字段说明：

- `wfParamName`
  - 工作流参数名
- `wfParamCnName`
  - 工作流参数中文名，可为空字符串
- `wfParamValue`
  - 工作流参数值

## 4. 返回结果

返回类型：`JSONResult<Long>`

顶层字段：

- `success`
- `msg`
- `data`

其中：

- `success = true` 表示接口调用成功
- `data` 为工作流实例 ID

成功返回示例：

```json
{
  "status": 200,
  "msg": "操作成功！",
  "success": true,
  "data": "301056192465629184"
}
```

失败返回示例：

```json
{
  "msg": "用户未登录",
  "code": 601,
  "success": false
}
```

## 5. 通用调用步骤

任意业务接入 BASE 工作流时，建议按以下步骤处理：

1. 准备业务主键，作为 `approvalObjectId`
2. 生成可读标题，作为 `approvalObjectTitle`
3. 明确业务类型编码，作为 `approvalObjectType`
4. 组织工作流参数列表 `approvalFormData`
5. 调用 `POST /workflow/submit`
6. 解析返回结果中的工作流实例 ID
7. 将工作流实例 ID 保存到业务主表
8. 在业务主表同步保存审批状态

## 6. 业务表推荐存储字段

建议在业务主表至少保留以下字段：

- `workflow_instance_id`
  - BASE 返回的工作流实例 ID
- `approval_status`
  - 当前审批状态

审批状态值建议按接入应用统一约定，例如：

- `UNSUBMITTED`：未提交
- `IN_PROGRESS`：审批中
- `APPROVED`：审批通过
- `REJECTED`：审批驳回

推荐处理方式：

- 发起前：`UNSUBMITTED`
- 提交成功后：更新为 `IN_PROGRESS`
- 审批通过后：更新为 `APPROVED`
- 审批驳回后：更新为 `REJECTED`

## 8. Java 调用示例

示例仅演示“发起工作流”的通用方法，不包含具体业务落库逻辑。

```java
public Long submitWorkflow(
        String baseUrl,
        String accessToken,
        Long wfBaseDefFileId,
        String approvalObjectId,
        String approvalObjectTitle,
        String approvalObjectType,
        List<Map<String, String>> approvalFormData
) throws IOException {
    URL url = URI.create(baseUrl + "/workflow/submit").toURL();
    HttpURLConnection connection = (HttpURLConnection) url.openConnection();
    connection.setRequestMethod("POST");
    connection.setDoOutput(true);
    connection.setRequestProperty("Accept", "application/json");
    connection.setRequestProperty("Content-Type", "application/json; charset=UTF-8");
    connection.setRequestProperty("Authorization", "Bearer " + accessToken);
    connection.setRequestProperty("Cookie", "seabox_acc_token=" + accessToken + "; access_token=" + accessToken);

    String body = """
            {
              "wfBaseDefFileId": %d,
              "approvalObjectId": "%s",
              "approvalObjectTitle": "%s",
              "approvalObjectType": "%s",
              "approvalFormData": [
                {
                  "wfParamName": "approveType",
                  "wfParamCnName": "",
                  "wfParamValue": "example_001"
                }
              ]
            }
            """.formatted(
            wfBaseDefFileId,
            approvalObjectId,
            approvalObjectTitle,
            approvalObjectType
    );

    try (OutputStream outputStream = connection.getOutputStream()) {
        outputStream.write(body.getBytes(StandardCharsets.UTF_8));
    }

    try (InputStream inputStream = connection.getInputStream()) {
        String response = new String(inputStream.readAllBytes(), StandardCharsets.UTF_8);
        // 按项目实际 JSON 工具解析 success / data
        return null;
    } finally {
        connection.disconnect();
    }
}
```

说明：

- 上面示例只演示请求方式
- 实际项目中应补齐 JSON 解析、异常处理、日志记录和返回值校验
- `approvalFormData` 应根据具体工作流定义动态组织，而不是固定写死

## 9. 审批页接口约定

当前“审批”页包含以下 3 个 tab：

- 我的待办
- 我的流程
- 已关闭流程

其中：

- 我的待办：调用 `POST /workflow/node/page`
- 我的流程：调用 `POST /workflow/pageMyWfInstance`
- 已关闭流程：也是调用 `POST /workflow/pageMyWfInstance`

这 3 个 tab 共同会带的关键查询参数是：

- `approvalObjectType`
- `keyWord`
- `pageNum`
- `pageSize`
- `tenantId`

说明：

- `approvalObjectType` 建议按业务域固定编码传递，并在具体项目中保持稳定
- `tenantId` 必须从当前登录用户信息中获取，不要写死
- 这几个查询接口均使用 `POST`，但不传请求体，参数统一拼到 query string

## 10. 查询我的流程

查询“我发起的流程”使用以下接口：

```http
POST /workflow/pageMyWfInstance
```

说明：

- 请求方法是 `POST`
- 无请求体
- 查询条件放在 URL query 参数中
- 调用时同样需要透传当前登录用户的 `Authorization` 和 `Cookie`

### 10.1 常用参数

审批页中的“我的流程”与“已关闭流程”这两个 tab，通用接入时通常关注以下字段：

- `keyWord`
  - 搜索关键字
- `approvalObjectType`
  - 审批对象类型
- `tenantId`
  - 当前租户 ID
- `pageNum`
  - 页码，从 `1` 开始
- `pageSize`
  - 每页条数

其他参数如 `filterType`、`approvalObjectId`、`approvalObjectTitle`、`appCode`、`moduleCode`、`sysClass`、`moduleClass` 可按具体业务需要补充。

补充说明：

- “已关闭流程”页在当前项目中同样调用 `POST /workflow/pageMyWfInstance`
- 是否视为“已关闭”，由前端或业务侧按返回状态自行归类展示

### 10.2 cURL 调用示例

```bash
curl -v -X POST 'http://leaf-base-server.dev.jinxin.cloud/workflow/pageMyWfInstance?approvalObjectType=example_business&keyWord=&tenantId=1167337764142780417&pageNum=1&pageSize=20' \
  -H 'Authorization: Bearer {access_token}' \
  -H 'Cookie: seabox_acc_token={access_token}; access_token={access_token}' \
  -H 'Accept: application/json'
```

### 10.3 前端调用示例

```ts
async function pageMyWfInstance(baseHost: string, accessToken: string, tenantId: string) {
  const query = new URLSearchParams({
    approvalObjectType: "example_business",
    keyWord: "",
    tenantId,
    pageNum: "1",
    pageSize: "20"
  });

  const response = await fetch(`${baseHost}/workflow/pageMyWfInstance?${query.toString()}`, {
    method: "POST",
    credentials: "include",
    headers: {
      Accept: "application/json",
      Authorization: `Bearer ${accessToken}`
    }
  });

  if (!response.ok) {
    throw new Error(`查询我的流程失败，状态码 ${response.status}`);
  }

  return response.json();
}
```

说明：

- `tenantId` 应优先从当前登录用户信息中获取，例如 `fetchMe().tenant.tenantId`

### 10.4 返回结果读取方式

根据 [03-Base服务接口说明.md](./03-Base服务接口说明.md)，该接口返回类型为 `PageResult<WfInstanceApiVO>`，常用顶层字段如下：

- `pageNum`
- `pageSize`
- `total`
- `data`

其中 `data` 为流程列表，列表项中常用字段包括：

- `wfInstanceId`
- `approvalObjectId`
- `approvalObjectTitle`
- `approvalObjectType`
- `approveStatus`
- `currNode`
- `currNodeName`
- `taskStartTm`
- `taskEndTm`
- `modifyTm`

前端表格渲染时，通常至少会读取：

- 流程实例 ID：`wfInstanceId`
- 标题：`approvalObjectTitle`
- 类型：`approvalObjectType`
- 当前节点：`currNodeName`
- 审批状态：`approveStatus`
- 发起或开始时间：`taskStartTm`

### 10.5 通用处理建议

查询“我的流程”时，建议统一遵循以下规则：

1. 请求方法固定使用 `POST`
2. 不传请求体，参数统一拼到 query string
3. `tenantId` 从当前登录用户信息获取，不要写死
4. 列表返回的 ID 类字段按字符串处理，避免前端长整型精度丢失

- `approvalObjectType`
- `keyWord`
- `pageNum`
- `pageSize`
- `tenantId`

### 11.2 cURL 调用示例

```bash
curl -v -X POST 'http://leaf-base-server.dev.jinxin.cloud/workflow/node/page?approvalObjectType=asset_inventory&keyWord=&tenantId=1167337764142780417&pageNum=1&pageSize=20' \
  -H 'Authorization: Bearer {access_token}' \
  -H 'Cookie: seabox_acc_token={access_token}; access_token={access_token}' \
  -H 'Accept: application/json'
```

### 11.3 返回结果常用字段

根据 [03-Base服务接口说明.md](./03-Base服务接口说明.md)，该接口返回类型为 `PageResult<WfInstanceNodeApiVO>`，常用字段包括：

- `wfInstanceId`

- `wfTaskTitle`
- `nodeName`
- `approvalObjectId`
- `approvalObjectTitle`
- `approvalObjectType`

### 11.4 待办审批动作接口

如果待办页需要提供“通过 / 驳回”按钮，对应 BASE 接口如下：

- 通过：`POST /workflow/node/complete`
- 驳回：`POST /workflow/node/reject`

这两个接口通常至少需要关注以下参数：

- `wfInstanceNodeId`
- `approveOpinion`
- `approveAttachmentUrl`

## 12. 常见问题

### 12.1 返回“用户未登录”

通常需要检查：

- `Authorization` 是否为标准格式 `Bearer {access_token}`
- `Cookie` 中是否带了 `seabox_acc_token` 和 `access_token`
- 当前 token 是否仍然有效

### 12.2 返回 HTTP 200，但 `success=false`

说明网关和接口已通，但业务校验未通过。应优先查看：

- `msg`
- `code`

### 12.3 返回工作流实例 ID 后，还需要做什么

至少还要做两件事：

- 将实例 ID 保存到业务主表
- 将审批状态更新为“审批中”
